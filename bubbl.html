bubbl! Anchored, real-time captions designed to stay with the speaker

bubbl! is an accessibility-focused mobile communication app for deaf and hard-of-hearing users that places live captions near the person speaking instead of in a static transcript bar. It combines camera input, speech streaming, face tracking, dynamic overlays, and context cues to make conversation feel more natural, readable, and socially connected.

Project Overview

bubbl! is an accessibility-focused mobile communication system designed for deaf and hard-of-hearing users. The app rethinks live captioning as a spatial interaction problem rather than a traditional transcription problem. Instead of presenting speech as subtitles in a static transcript area at the bottom of a screen, bubbl! places real-time captions visually near the person speaking. The goal is to reduce attention shifting, preserve eye contact, and make spoken conversation feel more natural, readable, and socially connected.

At a high level, the product combines real-time speech streaming, camera-based face tracking, dynamic caption rendering, contextual tone inference, session history, adaptive accessibility settings, and experimental alternative communication modes. Under the hood, it is a multimodal real-time system that has to coordinate audio capture, networked speech recognition, computer vision, overlay positioning, and animation smoothly enough to feel calm during actual human conversation.

This project has required me to think beyond the boundaries of a typical mobile UI implementation. The hard problem is not just displaying text. The hard problem is building a system that can continuously perceive a live environment, interpret rapidly changing audio and visual signals, and convert them into a stable, readable, non-intrusive interface that helps someone understand speech without pulling them away from the person they are talking to.

Problem Framing

Most live captioning tools optimize for text accuracy and transcript completeness, but not for the lived interaction of conversation. In practice, standard transcription interfaces force the user to look down at a fixed text region, separate from the speaker’s face. That can break conversational flow, reduce nonverbal context, and increase the cognitive load of simply following a person speaking in real time.

bubbl! starts from a different premise: the interface should follow the conversation, not force the conversation to adapt to the interface. That means the app has to:

  • listen to live speech with low enough latency that the text feels immediate
  • determine where the speaker is in the camera frame
  • anchor the caption bubble near that speaker instead of at the bottom of the screen
  • preserve legibility without blocking too much of the visual scene
  • keep previous context visible briefly without flooding the display
  • provide optional assistive cues about tone, emphasis, and inferred meaning
  • remain configurable enough to support different hearing needs, environments, and comfort levels

The project is therefore equal parts accessibility design, systems engineering, and real-time interface research.

Product Goal

The core product goal is simple to describe but difficult to engineer: if a deaf or hard-of-hearing user points their phone at someone speaking, the spoken words should appear near that person in a way that feels fast, stable, and visually natural.

That product goal breaks down into several engineering requirements:

  • Low latency so captions do not arrive too late to be conversationally useful
  • Spatial relevance so the text appears connected to the speaker
  • Visual stability so bubbles do not jitter or snap aggressively
  • Legibility so the overlay remains readable in live camera conditions
  • Context preservation so previous speech remains accessible without clutter
  • Graceful degradation so the app stays useful even when tracking or inference is imperfect
  • Accessibility control so users can customize the experience to their needs

Technical Stack

The Android implementation is being built natively in Kotlin using Jetpack Compose for the UI layer. The app uses:

  • CameraX for camera preview and analysis pipelines
  • ML Kit Face Detection for face tracking and speaker anchoring
  • Deepgram streaming speech recognition over persistent WebSocket for low-latency transcription
  • Room for transcript persistence and searchable history
  • DataStore for persistent accessibility and behavior settings
  • Custom audio analysis for volume estimation and emphasis cues
  • Hume emotion streaming for multimodal affective context experiments
  • MediaPipe-based gesture infrastructure for experimental ASL-related mode work

The architectural challenge is less about any individual tool and more about how all of these systems interact at the same time on a mobile device with constrained CPU, GPU, network, memory, and thermal budgets.

System Architecture

The application is built as a state-driven pipeline with a modular separation between capture, interpretation, persistence, and presentation.

1. Input Layer

The input layer collects data from several independent live sources:

  • camera preview frames
  • microphone PCM audio
  • streaming transcript updates
  • face detection signals
  • emotion model outputs
  • optional ASL/gesture preview data

Each of these input streams has very different timing characteristics. Speech recognition streams partial words rapidly. Face detections arrive frame-by-frame and can jitter. Emotion inference tends to be slower and more probabilistic. ASL analysis requires image processing windows rather than instantaneous events. The app architecture has to normalize these streams into a UI model that remains coherent under constant change.

2. Interpretation Layer

The interpretation layer converts raw sensor data into higher-level UI meaning:

  • speech audio becomes partial and final transcript text
  • face detections become target anchor regions in normalized screen space
  • volume samples become emphasis intensity and dynamic text styling cues
  • emotion model outputs become assistive labels such as warm, frustrated, joyful, or uncertain
  • history timestamps become grouped conversations rather than isolated sentence fragments

This layer is especially important because it acts as the bridge between noisy machine outputs and a calm user-facing experience.

3. Presentation Layer

The Compose presentation layer renders:

  • live camera preview
  • dynamic speech bubbles anchored near the speaker
  • trailing caption history that fades naturally
  • emotion/tone pills
  • debug overlays when needed
  • settings, history, and alternate input mode controls

The UI is intentionally minimalist and high contrast because the screen has to compete with real-world motion, lighting changes, and the limited time a user has to process information during live interaction.

Real-Time Speech Pipeline

One of the most important technical decisions in the project was moving away from end-of-speech mobile transcription patterns and toward a true streaming speech architecture.

Traditional mobile speech APIs often wait until the user pauses before they finalize useful text. For a conversational accessibility product, that delay is too expensive. Even a one to three second lag can make the interface feel disconnected from the speaker. To solve that, bubbl! uses a persistent streaming transcription path where audio is captured in small chunks and sent continuously, allowing partial transcripts to appear rapidly and update in place.

This required several pieces of engineering work:

  • replacing stop-and-restart speech loops with a persistent WebSocket model
  • chunking microphone audio into small enough units for responsiveness while avoiding excessive overhead
  • rendering partial transcript updates smoothly rather than as large block replacements
  • deciding when to commit partial text into finalized caption history
  • handling silence, reconnects, and network instability gracefully
  • ensuring the pipeline can recover automatically without the user feeling the failure mode directly

In practice, the UI treats partial and final text differently. Partial text is intentionally presented as provisional, lower-stability speech content, while finalized text becomes part of the persistent conversation log. That distinction is critical because it lets the user follow speech in real time without confusing unstable interim words with final content.

Bubble Rendering and Text Motion

The speech bubble system ended up being one of the most nuanced parts of the project.

At first glance, a speech bubble sounds like a simple overlay. In reality, the bubble has to solve several competing requirements at once:

  • expand quickly enough to match incoming words
  • avoid abrupt jumps in size
  • stay visually attached to the speaker
  • remain readable over complex backgrounds
  • avoid covering too much of the face
  • fade away naturally when conversation pauses
  • preserve conversational context by carrying older captions briefly

To make the text feel less mechanical, the app does not simply replace full strings in a hard snap. Instead, the bubble is tuned to reveal and settle text in a more intentional, fluid way. The size adjustment is animated, horizontal scaling is allowed as the line grows, and old captions drift upward and fade rather than disappearing instantly. This helps the interface feel closer to a live conversational artifact than a typical subtitle renderer.

There is also damping and recoil built into movement so the bubble follows camera motion with more of a stabilized, gimbal-like character rather than sticking rigidly to raw detection coordinates. That small difference matters a lot perceptually. A purely literal tracker feels jittery and robotic. A slightly smoothed tracker feels more human-centered and legible.

Face Tracking and Spatial Anchoring

Spatial anchoring is central to the entire product concept. The app uses the camera not just as a visual background, but as a way of understanding where speech should appear.

The current implementation uses face detection to estimate where the main speaker is in normalized screen coordinates. That anchor is then translated into an ideal bubble position above or near the face, and the UI applies:

  • edge clamping so the bubble stays on screen
  • positional smoothing to reduce jitter
  • spring behavior to make movement feel natural
  • fallback placement when no face is confidently available

This is technically challenging because camera detections are inherently noisy. People move unpredictably. Lighting changes. A face may partially leave the frame. A bubble cannot simply lock to raw bounding boxes frame-by-frame or it will vibrate visually.

As a result, the app treats face position as a target rather than a literal absolute. The overlay is always interpolating toward that target while respecting screen boundaries and visual clarity.

This part of the architecture is also what makes the project naturally extensible toward future AR and wearable computing.

Emotion and Context Inference

Plain transcription removes a large amount of social information from speech. Someone can say the same sentence warmly, sarcastically, aggressively, or nervously, and the transcript alone may not communicate that difference. To explore how assistive context might supplement text, I built an experimental multimodal emotion layer.

That system does not attempt to be authoritative or perfect. Instead, it aims to provide lightweight, clearly assistive signals based on multiple sources:

  • facial expression analysis
  • vocal prosody and intensity
  • transcript-level textual cues

These inputs are fused into a small set of practical display categories such as warm, joyful, frustrated, sad, fearful, surprised, confused, or neutral. The interface is intentionally cautious about confidence and is designed so that the app remains fully usable even if emotion inference is disabled or low confidence.

From an engineering perspective, this emotion layer introduced a second class of real-time complexity:

  • managing another persistent streaming connection
  • combining asynchronous face and audio emotion responses
  • preventing low-signal outputs from overwhelming the UI
  • making sure the assistive tone indicator is visually subtle enough to remain supportive, not distracting

This work reflects a broader theme in the project: the most interesting engineering problem is often not model integration itself, but deciding how model outputs should be filtered, fused, timed, and displayed so they are actually useful to a person in the middle of a live conversation.

Accessibility and Settings Architecture

Because this app is built for accessibility use cases, configuration is not a nice-to-have feature. It is a core architectural requirement.

Different users need different balances between density, speed, emphasis, opacity, and context. Someone in a noisy café may want stronger visual cues and slower fade behavior. Someone in a quiet one-on-one conversation may want a lighter overlay with minimal visual intrusion.

The settings system therefore exposes controls for:

  • text size
  • bubble opacity
  • fade duration
  • emphasis intensity
  • emotion indicator visibility
  • inferred text expansion
  • debug overlays
  • conversation grouping thresholds
  • anchoring mode selection

These values persist across sessions and flow directly into the live UI so changes can be felt immediately rather than requiring restarts or hidden configuration.

The accessibility mindset has also shaped visual design choices:

  • dark, high-contrast bubble surfaces
  • large tap targets
  • readable overlay typography
  • minimal decorative noise
  • constrained visual density on the camera screen
  • support for debug and fallback states so users can understand what the system is doing

Transcript History and Conversation Grouping

Another major iteration in the project involved moving from sentence-level transcript storage toward more meaningful conversation-level history.

Raw streaming transcripts naturally arrive as fragmented utterances. If stored one by one, the resulting history becomes noisy and hard to interpret. To make the feature useful, the app groups temporally related transcript entries into conversation sessions. If new speech arrives after a sufficiently long gap, the next block is treated as a new conversation rather than just another line in the same session.

That shift matters for usability because history should reflect how people remember conversations, not how APIs emit transcription events. A user wants to revisit a discussion, not scan through a long list of disconnected sentence fragments.

This required work on:

  • timestamp-aware grouping logic
  • adjustable session gap thresholds
  • history search and export behavior
  • keeping ephemeral on-screen fades separate from durable persisted records

Experimental ASL / Visual Communication Mode

In addition to speech-to-text, I’ve also explored an experimental camera-based sign language and gesture interpretation mode. This is still early and clearly labeled as such, but it represents an important extension of the product philosophy: accessibility should not assume a single communication direction.

Engineering-wise, that introduced a very different operating mode from the speech pipeline. Instead of relying on microphone and speech recognition, the app has to watch the camera feed for gesture-like signals and translate them into readable output. That mode raised significant performance questions because visual inference is resource intensive, especially when running on-device alongside preview, overlays, and optional emotion processing.

To keep that path viable, I had to:

  • reduce redundant image analysis use cases
  • throttle preview sampling
  • resize inputs strategically
  • disable or pause other expensive live systems when ASL mode is active
  • separate optimized builds for devices with tighter resource budgets

This work reinforced an important systems lesson: adding new intelligence to a real-time mobile product is not just a matter of model integration. Every new analysis path competes for the same limited resources, so feature design and performance engineering have to happen together.

Performance, Stability, and Build Strategy

The app has grown into a system with multiple simultaneous real-time subsystems:

  • live camera preview
  • face detection
  • streaming speech transcription
  • audio analysis
  • emotion inference
  • animated overlay rendering
  • session persistence
  • optional ASL interpretation

On mobile hardware, especially across different devices, that creates real pressure on CPU, memory, thermal load, and camera/audio concurrency. A large amount of the engineering effort has gone into keeping those subsystems from stepping on each other.

Some of the problems I’ve had to solve include:

  • camera startup failures when too many use cases are bound at once
  • crashes from competing microphone ownership paths
  • resource contention when ASL, emotion, and speech systems all run together
  • instability when switching front and back cameras
  • balancing overlay fidelity with rendering smoothness
  • handling degraded or absent model outputs without breaking the experience

I also introduced an optimized app variant so heavier experimental modes can be tuned more conservatively on weaker devices without sacrificing the main experience. That build strategy reflects a practical reality of mobile accessibility products: if the experience is powerful but unstable, it is not actually accessible.

Design Philosophy

The design direction for bubbl! is intentionally restrained. The app is not trying to impress with spectacle. It is trying to disappear just enough to support communication.

That philosophy has guided a number of product decisions:

  • keep the interface mobile-first and uncluttered
  • treat context signals as assistive, not authoritative
  • preserve as much eye contact and face visibility as possible
  • avoid decorative UI that competes with the real world
  • show system state clearly when confidence or permissions are limited
  • favor fluidity and stability over raw visual novelty

In that sense, the project is not just about AI or AR. It is about interaction design under cognitive load. The real test of the system is not whether a feature works in a demo, but whether it helps someone stay present in a conversation.

What Makes This Project Technically Interesting

There are several reasons I view bubbl! as one of the most interesting engineering projects I’ve worked on.

It is inherently multimodal.

The app is constantly reasoning across audio, video, text, motion, and UI state. Most projects let one modality drive the interface. bubbl! has to synthesize several.

It is strongly real-time.

The user experience depends not just on correctness, but on timing. Even a correct output can feel broken if it arrives too late, animates poorly, or jitters.

It is accessibility-first.

The system is not built around novelty. It is built around reducing friction in a real communication problem, which raises the bar for reliability and design quality.

It requires graceful failure.

Every perception system is noisy. The product only works if incomplete tracking, imperfect emotion inference, or variable network quality still degrade into a useful experience.

It creates a direct bridge to wearable computing.

The spatial caption model developed on the phone is already conceptually close to a heads-up, glance-preserving wearable display system.

Planned Next Steps

The current version is already tackling real-time captioning, face anchoring, history, settings, emotion cues, and experimental alternate communication modes, but the product roadmap is much larger.

Multi-speaker support

One of the next major steps is support for multi-speaker conversations. The long-term goal is for the app to distinguish multiple people in frame, assign transcription segments to different speakers, and render separate bubbles that track each person without overlapping. That requires a deeper fusion of diarized speech output, multi-face tracking, and layout conflict resolution. It also raises interesting product questions around emphasis styling, overlap avoidance, visual hierarchy, and how much simultaneous context a user can reasonably process.

Better anchored AR interaction

The current phone-based overlay system already behaves like an AR-style interface, but there is room to improve how world-locked the bubble feels under movement and how robust the anchor remains as the user changes viewpoint. Future versions can push further into true anchor persistence, scene-aware placement, and more stable conversational overlays.

Stronger sign language support

The ASL-related mode is still experimental. Longer term, I want to move from limited gesture recognition toward richer sign understanding, better temporal modeling, and more realistic deaf-first interaction patterns. That will require much more specialized model and dataset work.

Multi-device and wearable extension

The project is intentionally being built in a way that can evolve beyond a handheld phone app.

AR Glasses Compatibility Roadmap

One of the most exciting future directions for bubbl! is AR glasses compatibility.

The current mobile app is the right first platform because it allows rapid iteration on capture pipelines, spatial overlays, interaction design, and accessibility settings. But the deeper product vision extends beyond the phone. The most natural version of this experience is one where captions appear in the user’s field of view while they maintain direct visual attention on the speaker.

That future system would preserve the same underlying principles:

  • low-latency real-time speech-to-text
  • speaker-anchored visual placement
  • contextual tone and emphasis cues
  • minimal visual clutter
  • accessibility-first customization

However, moving from a phone to glasses creates a very different technical environment.

1. Anchor stability becomes much more important.

On a phone, a little bubble movement can be acceptable as long as it remains readable. On a wearable display, unstable anchors would be much more noticeable and more fatiguing. That means the system will need stronger head-pose compensation, better spatial smoothing, and more robust methods for keeping captions visually attached to a speaker in a way that feels physically coherent.

2. World-space UI matters more than screen-space UI.

The current implementation largely thinks in screen coordinates because the phone display is the rendering surface. A glasses-based version would need to think in persistent world-space or head-relative coordinates. That changes how anchors are represented, updated, and stabilized.

3. Power and compute budgets become tighter.

Wearable devices have stricter constraints than phones. That means every subsystem has to be reconsidered in terms of:

  • local vs remote inference
  • streaming bandwidth
  • frame-rate targets
  • thermal and battery impact
  • sensor fusion strategy

4. UI density needs to be rethought.

A phone gives the user a large, explicit screen. Glasses do not. Overlay density, caption length, fade timing, and the number of simultaneous on-screen elements all become much more sensitive. The app will need stronger prioritization rules about what should appear, where, for how long, and with what visual weight.

5. Interaction models change.

On a phone, the user can tap settings, toggle modes, zoom the camera, and navigate through screens. On glasses, those interactions may need to become voice-driven, phone-companion-driven, or gesture-based. That means the mobile app I’m building now is not only a product in itself, but also likely the control surface for a future wearable version.

6. Social ergonomics matter even more.

If the long-term goal is to support natural conversation, wearable rendering has to feel subtle and trustworthy. The interface cannot flood the user with unstable, overconfident, or noisy cues. This is as much a product design challenge as a technical one.

The reason I am excited about this direction is that bubbl! already solves the right foundational problems for it:

  • real-time speech streaming
  • spatially anchored caption logic
  • dynamic overlay timing
  • context inference
  • accessibility personalization

The phone implementation is not separate from the AR glasses vision. It is the proving ground for it.

Why This Project Matters to Me

bubbl! represents the kind of engineering work I care most about: technically ambitious systems built around a real human need.

It has forced me to work across domains instead of staying inside one comfort zone. I’ve had to think like a mobile engineer, interface designer, real-time systems developer, and product designer at the same time. I’ve had to make tradeoffs between capability and clarity, between responsiveness and stability, and between experimental intelligence and dependable user experience.

Most importantly, it has pushed me to think seriously about what it means for software to help someone stay connected in a moment that matters. The value of the project is not in having a clever model stack. The value is in making communication feel more accessible, more immediate, and more humane.
