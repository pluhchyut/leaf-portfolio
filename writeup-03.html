Case Study 03 • Heap / Pwn

Heapedit: reading glibc tcache like a data structure, not a mystery

Flag recovered • adapted from the five-case credibility essay
Binary Exploitation Allocator Internals Tcache Case Study

Challenge Summary

The flag was never behind shellcode or a return-address overwrite. It was sitting in a freed heap chunk, and the solve depended on recognizing that the allocator’s free-list pointers were the real structure to follow.

Why It Was Hard

This kind of pwn problem punishes guessing. The only clean path is to understand how a freed chunk stores the next pointer and how the program validates the traversal it expects back.

Approach

  • Read the source all the way through. In heapedit.c, the program allocates six same-sized chunks, zeros them, reads the flag from disk, and copies it into the last chunk at FLAG_OFFSET, right after the pointer-sized region that tcache uses for free-list metadata.
  • Pay attention to free order. The chunks are freed in reverse order, which makes the final tcache head chunks[0]. The binary even prints that head, turning the entire task into a linked-list reconstruction problem.
  • Follow allocator metadata, not the flag text. After each correct answer, the program copies the next pointer out of the freed chunk into next. Once I saw that, the solve collapsed into free-list order rather than fuzzing random addresses.
  • Use the invariant to recover the final chunk. Because all six chunk addresses are known and the program validates that each next pointer belongs to that set, the traversal can be reasoned through cleanly.
  • Let the program print the reward. After the traversal is satisfied, the binary prints chunks[CHUNK_COUNT-1] + FLAG_OFFSET. The exploit is really about proving you understand allocator state well enough for the binary to reveal the string itself.

This is one of my favorite examples of how source-reading turns “heap sorcery” into a very understandable data-structure problem.

Source Snapshot

memcpy((char*)chunks[CHUNK_COUNT-1] + FLAG_OFFSET, flag_buf, flen + 1);
void *head = chunks[0];
printf("tcache head (start of free list) -> %p\n", head);

for (int i = CHUNK_COUNT - 1; i >= 0; --i) {
    free(chunks[i]);
}

Read the allocator layout first.
The exploit path is a traversal problem, not a smash-the-stack problem.

Outcome

Flag recovered in the original solve by reconstructing the free-list order correctly and following allocator metadata all the way to the chunk that still held the copied flag bytes.

The broader lesson is that many hard heap challenges become much cleaner once I stop thinking in terms of corruption first and start reasoning in terms of allocator invariants.
