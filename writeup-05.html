Case Study 05 • Cloud / Infrastructure

Kubernetes secrets: turning cluster access into the flag with almost no app logic

Flag recovered • adapted from the five-case credibility essay
Cloud / Infra Kubernetes Secrets Enumeration Case Study

Challenge Summary

This case is a reminder that not every compromise looks like code execution. Sometimes the real path is simply that the control-plane credentials have already been handed over and the rest is calm, correct enumeration.

Why It Was Hard

The tricky part was not exploitation syntax. It was realizing that the supplied kubeconfig was the attack surface, and that Kubernetes secrets are only “hidden” until someone with the right identity asks the control plane for them.

Approach

  • Start with the credentials you were given. kubeconfig.yaml already contained the cluster information, client certificate, and private key material needed to authenticate.
  • Make the config usable against the cluster. Once the challenge endpoint was live, I used the supplied kubeconfig with kubectl and followed the hint about relaxed TLS handling just to prove I could talk to the control plane as the provided identity.
  • Enumerate before guessing object names. I started with kubectl get namespaces and kubectl get secrets -A instead of inventing object names or assuming the flag lived in default without evidence.
  • Pull the secret as structured data. Once the right secret object was identified, I fetched it with JSON or jsonpath. At that point the remaining step was base64 decoding, not “exploitation” in the flashy sense.
  • Treat the platform itself as the target. There was no need for code execution, pod escape, or container abuse. Authenticated access to the Kubernetes API already gave the correct read path.

This writeup is in the portfolio because it shows a calmer style of security work: understand the platform, trust the data model, and let enumeration do the heavy lifting.

kubectl Snapshot

$ kubectl --kubeconfig kubeconfig.yaml get secrets -A
NAMESPACE   NAME          TYPE     DATA
default     flag-secret   Opaque   1

$ kubectl --kubeconfig kubeconfig.yaml get secret flag-secret -o jsonpath='{.data.flag}'
cGljb0NURnt...base64...

$ python3 -c 'import base64; print(base64.b64decode("...").decode())'
flag material recovered from the control plane itself

Outcome

Flag recovered in the original solve by authenticating with the supplied kubeconfig, enumerating the relevant secret, and decoding the value directly from the control plane.

Infrastructure challenges often reward patience over aggression. This page is here because it shows that calm enumeration can be the entire exploit path when the platform model is the real challenge.
