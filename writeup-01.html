Case Study 01 • Web Exploitation

Paper-2: browser exploit research under a locked-down bot

Research case / no flag • adapted from the five-case credibility essay
Web Exploitation Research Case CSP CSS Oracle Case Study

Challenge Summary

This challenge stopped being a normal web problem almost immediately. The real task was not “find XSS,” but figuring out what a bot browser would still do once CSP, URL allowlists, and one-shot secrets were stacked on top of each other.

Why It Was Hard

The app exposed the secret in body[secret], but the flag endpoint used getdel, so every wrong request destroyed the only live token. External navigation was blocked, JavaScript exfiltration was unreliable, and several promising browser primitives turned into dead ends on contact.

I kept this case in the portfolio even without a flag because it shows the exact kind of measured debugging and environment modeling that harder web challenges demand.

Approach

  • Read the server, not the challenge copy. I mapped the request flow directly from the application: uploads lived at /paper/:id, the bot only visited uploaded files, /secret reflected a fresh token into both text and a body attribute, and /flag?secret=... burned the token with getdel on every bad request.
  • Break the problem into browser capabilities. Instead of aiming straight for a flag steal, I reduced the challenge into smaller questions: can the bot redirect, submit forms, turn DOM attributes into URLs, or trigger requests through prerender, CSS, or XSLT?
  • Test one primitive at a time. Tiny HTML, XML, and CSS probes revealed the useful split: external CSS would load but not build the same URL path as inline CSS, prerender gave some GET-side effects, and several POST-flavored exfil ideas never landed.
  • Stop trusting ambiguous signals. Responses like nice try were not enough proof on their own because bot startup failures or mistimed requests could burn the secret too. I tightened every experiment until only one moving part changed at a time.
  • Narrow to the smallest viable oracle. The best line of work became inline CSS, a minimal payload, and incremental recovery based on browser behavior that repeated cleanly instead of looking clever once.

The core lesson was to turn a chaotic browser challenge into a shortlist of real, measurable primitives and ignore everything that only looked exciting once.

Lab Snapshot

Test 1  External CSS + if(style()) loaded the stylesheet, but never hit /flag
Test 2  Inline CSS + if(style()) did trigger the right request path
Test 3  Prerender JS allowed some GET side effects, but browser-side POST exfil stayed dead
Test 4  attr(secret) + image-set() became the interesting direction
Conclusion  Treat the bot like a hostile runtime and turn it into a small, repeatable oracle

Outcome

Research case only. No flag was recovered here, but the investigation produced a reliable map of which browser primitives were real, which ones were fake wins, and how close the oracle path actually came to a usable exploit.

It remains one of the strongest examples in the portfolio of how I cut weak ideas quickly and keep reducing a hostile environment until the remaining questions are precise.
