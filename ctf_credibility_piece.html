Case 01

Paper-2: browser exploit research under a locked-down bot

Research Case / No Flag Web Exploitation Hardest Research Case CSP / CSS Oracle

This one stopped being a normal web challenge very quickly. The job was not “find XSS.” The job was understanding exactly what a bot browser would still do after CSP, URL allowlists, and one-shot secrets were layered on top of each other.

Why It Was Hard

The app exposed the secret in body[secret], but the flag endpoint used getdel, so every wrong guess destroyed the only live token. External navigation was blocked, JS exfil was unreliable, and a lot of seemingly promising browser primitives turned out to be dead ends.

I did not get the flag on this one. I am keeping it here because the investigation itself is one of the best examples of how I test assumptions, cut weak ideas, and turn browser behavior into a measured attack surface.

Step By Step

  1. Read the server, not the challenge copy. I started by mapping the real request flow from the application itself. Uploads were stored in Redis as /paper/:id, the bot only visited those uploaded files, /secret reflected a fresh 32-hex token into the page as both text and a body attribute, and /flag?secret=... used getdel, which meant every bad request burned the current token immediately.
  2. Break the problem into browser capabilities. Instead of asking “how do I steal the flag,” I asked much smaller questions: can the bot be redirected, can it submit a form, can it make a GET with attacker-controlled bytes, can CSS turn a DOM attribute into a URL, can XSLT fetch internal pages, and can prerender trigger requests that normal page flow would block?
  3. Test one primitive at a time. I uploaded tiny HTML, XML, and CSS probes and watched which ones changed the server state. That is where the important split showed up: external CSS loaded, but did not give me the same URL-building behavior that inline CSS did; prerender gave me some browser-side side effects, but not the write primitive I wanted; several POST-style exfil ideas simply never landed.
  4. Stop trusting ambiguous signals. A response like nice try looked exciting at first, but it was not good enough as proof because bot startup failures or mistimed requests could also consume the secret. I tightened the experiments so each test only changed one moving part and I used timing plus very early probes to separate real browser execution from noise.
  5. Narrow to the smallest viable oracle. The best direction was no longer “force the whole exfil in one payload.” It became: keep the payload inline, keep the CSS small, use the secret already exposed in the DOM, and recover state incrementally. I did not finish the flag path, but I did get the challenge down to a much smaller and more measurable browser-behavior problem.

The important part of this writeup is not a final flag. It is the workflow: reduce a chaotic browser challenge to a list of concrete primitives, kill the ones that lie, and only build on the ones that keep behaving the same way twice.

What I Learned

Hard browser exploitation is mostly measurement. The best progress came from logging what the environment would and would not do, then cutting away the “clever” payloads that looked exciting but were not stable.

Outcome

Research case only. No flag was recovered here, and I am keeping it in the portfolio because the investigative process is still one of the strongest examples of how I work through a hostile environment. The useful deliverable from this one was a reliable map of which browser primitives were real, which ones were fake wins, and how close the oracle path actually got.

Case 02

Router: turning leaked output and bad indices into a heap exploit

Binary Exploitation No Source Heap / OOB Index

This challenge looked like a harmless route planner at first. The real problem was understanding how order objects, map data, and rerouting logic were laid out on the heap when the only interface was a text menu.

Why It Was Hard

No source code, instance-specific binaries, and a bug hidden behind user-facing indices. The hints were honest: output leaked internal state, and indices that crossed boundaries did weird things on the heap.

Step By Step

  1. Learn normal behavior before touching the bug. I started with help, loaded each city map, created a single order, rerouted it normally, and dispatched it. That gave me the baseline output for a healthy order and made it obvious which fields changed when the program recalculated a path.
  2. Use order creation as heap grooming. Adding multiple orders was not just feature use; it was a way to force the binary to allocate a repeatable set of objects. By changing the number of live orders and rerunning the same commands, I could watch when route data started echoing extra bytes and when neighboring structures appeared to affect one another.
  3. Probe the index boundary carefully. The hints pointed straight at indices, so I checked valid values first, then values just outside the expected range, then negative indices. The useful moment was not “it crashed,” it was “it printed something that looked like internal state,” because that meant the bug was crossing object boundaries without immediately killing the process.
  4. Turn the leak into a map of adjacent heap state. Once the output started including data that did not belong to the selected order, I treated that as orientation. I repeated the same allocation pattern until I got similar leaks, then used those leaks to estimate which object sat next to which and what kind of field I was likely corrupting.
  5. Promote the out-of-bounds access into control. With the layout stabilized, the bug stopped being “weird output” and became a constrained write into adjacent route/order metadata. From there, the exploit path was to redirect behavior toward the hidden success logic rather than trying random large corruptions and hoping for a win.

The reason this solve mattered to me is that it was not a one-command trick. I had to make the binary teach me its own layout through repeated, disciplined interaction, then use the leak to turn an unreliable bug into a deliberate exploit.

What I Learned

When source is missing, the command interface becomes the debugger. The fastest way forward is to make the program explain its memory model through its own behavior instead of trying to “out-clever” it on the first pass.

Outcome

Flag recovered in the original solve after stabilizing the out-of-bounds heap behavior with leaked state from the text interface. Public version redacts the token as picoCTF{[redacted]}, but the important part is that the exploit path was repeatable, not a lucky one-off corruption.

Case 03

Heapedit: reading glibc tcache like a data structure, not a mystery

Binary Exploitation Allocator Internals Tcache

The flag was never hidden behind shellcode or a return address overwrite. It was sitting in a freed heap chunk, and the whole challenge was about knowing how tcache threads those chunks together.

Why It Was Hard

This is the kind of pwn problem that punishes guessing. You only get the right answer if you understand how a freed chunk stores the next pointer and how the program is validating the traversal.

Step By Step

  1. Read the source all the way through. In heapedit.c, the program allocates six same-sized chunks, zeros them, reads the flag from disk, and copies it into the last chunk at FLAG_OFFSET. That offset matters: the first pointer-sized bytes of a freed chunk are where the allocator stores the next free-list pointer, so the challenge author intentionally placed the flag just after that.
  2. Pay attention to free order. The code frees the chunks in reverse order, from chunks[5] down to chunks[0]. Because tcache is LIFO, the final head of the free list becomes chunks[0], which the program even prints for you before asking questions. That immediately turns the task into a linked-list reconstruction problem.
  3. Follow the allocator metadata, not the flag text. After each correct address, the binary does a memcpy of the first pointer-sized bytes from that freed chunk into next. In other words, it is literally traversing the tcache singly-linked list. Once I understood that, the answers were the chunk addresses in free-list order, not guesses about where the flag string might be.
  4. Use the invariant to recover the final chunk. Because all six allocations are known and the program validates that each next pointer is one of those known chunks, I could reason cleanly about the traversal instead of fuzzing it. The last chunk in that walk is the one whose user area still contains the flag bytes starting one pointer-width in.
  5. Let the program print the reward. After the traversal is satisfied, the binary prints chunks[CHUNK_COUNT-1] + FLAG_OFFSET. So the exploit was really about proving I understood the allocator state well enough for the binary to reveal the string on its own.

This is a good example of why source-reading matters. The challenge looks like heap sorcery from the outside, but in the source it collapses into a very understandable question about tcache layout and freed-chunk metadata.

What I Learned

A lot of “hard” heap challenges become much cleaner once I stop thinking in terms of corruption first and start thinking in terms of allocator invariants. Here, understanding the free list was the exploit.

Outcome

Flag recovered in the original solve by reconstructing the free-list order correctly and following the allocator metadata all the way to the chunk holding the copied flag bytes. Public version redacts the token as picoCTF{[redacted]}.

Case 04

VulnBank: draining a contract that “checked balances” but paid too early

Blockchain Smart Contract Audit Reentrancy

This is the classic pattern that keeps surviving code reviews: the check exists, the balance exists, and the contract still loses because the state update happens after the external call.

Why It Was Hard

The bug is conceptually simple, but real credibility comes from spotting it fast in source, turning it into a minimal attack contract, and understanding why the exploit works despite the presence of a balance check.

Step By Step

  1. Start with the order of operations. In VulnBank.sol, the vulnerable line is not subtle: msg.sender.call{value: amount}("") happens before balances[msg.sender] -= amount. That means the contract hands control to the recipient while its own accounting still reflects the old balance.
  2. Check the full win condition. I did not stop at spotting reentrancy. I also read the end of withdraw() and saw that the flag is only revealed when address(this).balance == 0. That changes the exploit goal from “prove I can reenter” to “drain the whole vault cleanly.”
  3. Write the smallest possible attack contract. The helper only needed a function to deposit a seed amount and a receive() fallback. In that fallback, I called back into withdraw() again while the previous call had not yet reduced balances[msg.sender]. Solidity 0.6-style low-level call forwards enough gas for that reentry path to stay alive.
  4. Use a controlled recursion amount. I seeded the bank with a small deposit and withdrew in the same increment repeatedly. Because the mapping check kept seeing the pre-decrement balance on each nested entry, the bank would keep paying out until its own ether balance approached zero.
  5. Stop at the reveal point, not after it. The clean solve is the one that empties the contract and triggers the reveal logic without overshooting or breaking the recursion. Once the vault balance hit zero, the contract emitted the flag and getFlag() became readable.

What made this satisfying was that the exploit was minimal. No fancy EVM trick was needed; just careful reading, one small receiver contract, and making the contract’s own payout order work against it.

What I Learned

“We added checks” is not a security argument. In smart contracts especially, the order of effects matters as much as the existence of a guard.

Outcome

Flag recovered in the original solve by using a reentrant receiver contract to withdraw repeatedly before the balance ledger updated, draining the vault to the exact reveal condition. Public version redacts the token as picoCTF{[redacted]}.

Case 05

Kubernetes secrets: turning cluster access into the flag with almost no app logic

Cloud / Infra Kubernetes Secrets Enumeration

This challenge was a good reminder that not every compromise looks like code execution. Sometimes the entire path is “you already have the control-plane credentials, so use them correctly.”

Why It Was Hard

The hard part was not exploitation syntax. It was recognizing that the supplied kubeconfig was the attack surface, and that Kubernetes secrets are only “hidden” until somebody with the right config asks for them.

Step By Step

  1. Start with the credentials you were given. kubeconfig.yaml already contains everything that matters: cluster information, a client certificate, and the private key material. The first lesson here was that the config file itself was the foothold, not just a setup artifact.
  2. Make the config usable against the exposed cluster. Once the challenge endpoint was live, I used the supplied kubeconfig with kubectl and ignored TLS validation issues per the challenge hint. The goal at this stage was simply to authenticate successfully and prove I could talk to the control plane as the provided identity.
  3. Enumerate the cluster before guessing names. The first commands were not “get the flag.” They were kubectl get namespaces, kubectl get secrets -A, and then more targeted requests once I knew which namespace and object names looked relevant. That kept me from wasting time on fantasy object names or assuming the flag lived in default without evidence.
  4. Pull the secret as structured data. Once the right secret object was identified, I fetched it as JSON or with a jsonpath expression. Kubernetes secrets are just base64-encoded key-value data, so the remaining step was decoding the field that held the flag rather than “exploiting” anything in the traditional sense.
  5. Treat the platform itself as the target. There was no need for pod escape, code execution, or container abuse. The solve was understanding that authenticated access to the Kubernetes API already gave me the exact read path I needed, and the flag fell out of standard cluster mechanics.

This writeup is here because it shows a different kind of security thinking. A lot of challenges want flashy exploitation; this one rewarded patience, enumeration, and trusting the platform’s data model.

What I Learned

Infrastructure challenges reward calm enumeration. The fastest route is usually not “attack harder,” it is “understand what object store, auth model, and serialization format you are already allowed to touch.”

Outcome

Flag recovered in the original solve by authenticating with the supplied kubeconfig, enumerating the right secret object, and decoding the stored value from the control plane. Public version redacts the token as picoCTF{[redacted]}.
