VulnBank: draining a contract that “checked balances” but paid too early

Flag recovered • adapted from the five-case credibility essay
Blockchain Smart Contract Audit Reentrancy Case Study

Challenge Summary

This is the classic smart-contract pattern that survives reviews more often than it should: the contract has checks, the balances exist, and it still loses because the external call happens before the state update.

Why It Was Hard

The bug is conceptually simple, but a strong writeup shows more than naming “reentrancy.” The real work is spotting it quickly in source, reading the actual win condition, and building the smallest exploit that drains the vault cleanly instead of just proving recursive entry exists.

Approach

  • Start with the order of operations. In VulnBank.sol, msg.sender.call{value: amount}("") executes before balances[msg.sender] -= amount, which means control is returned to the recipient while the old balance is still recorded.
  • Check the full win condition. The contract only reveals the flag when address(this).balance == 0, so the real goal is not just “reenter once,” but drain the vault to the exact reveal point.
  • Write the smallest possible attack contract. The helper only needed a seed deposit function and a receive() fallback that called back into withdraw() before the previous call had reduced the recorded balance.
  • Use a controlled recursion amount. By withdrawing in the same increment repeatedly, the mapping check kept seeing the old balance on nested entry and the vault kept paying out until it neared zero.
  • Stop at the reveal point. The clean solve is the one that empties the contract and unlocks the flag logic without overshooting or breaking the recursion.

What made this satisfying was how small the exploit was. No EVM trickery, just careful source reading and one receiver contract that turned the contract’s own payout order against it.

Source Snapshot

function withdraw(uint amount) public {
    require(balances[msg.sender] >= amount, "Insufficient funds available");

    (bool sent, ) = msg.sender.call{value: amount}("");
    balances[msg.sender] -= amount;

    require(sent, "Transfer failed");
}

External interaction before effect update = reentrancy window.

Outcome

Flag recovered in the original solve by using a reentrant receiver contract to withdraw repeatedly before the balance ledger updated, draining the vault to the exact reveal condition.

This case is in the portfolio because it shows that “we added checks” is not a security argument. In contracts especially, the order of effects matters just as much as the existence of a guard.
