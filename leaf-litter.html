Abstract

Capture the Flag competitions are not one problem. They are a rotating stack of entirely different disciplines forced into the same clock. A reversing challenge wants one workflow. A kernel pwn challenge wants another. A JWT bug, a lattice attack, a damaged QR code, a memory dump, and a weird mixed-category challenge all demand different instincts, different tooling, and different failure recovery paths.

That is exactly why systems like Squid Agent mattered so much to me. The big idea was not simply "use agents for CTFs." The real idea was that specialized agent systems can outperform generic loops because CTF categories are fundamentally different problem spaces.

Leaf Litter started as my own answer to that idea, but it did not stay small for long. What began as a solver concept turned into a much larger system: a challenge normalizer, a capability detector, a fast-path Codex wrapper, a generated category-specific multi-agent runtime, a live control center, a benchmark harness, a platform solver, a replay engine, a memory layer, a RAG layer, a self-healing execution layer, and a reporting/snapshot pipeline.

So this post is not a lightweight "I made a cool CTF agent" writeup. It is a deeper look at what Leaf Litter actually is, how heavily it was inspired by Squid Agent, how I built it, why I think it is genuinely a major step forward, and what I plan to add next.

The Core Idea

Leaf Litter exists because I do not think the interesting problem is "can an LLM solve a challenge sometimes?"

That question is already too small.

The real question is:

Can you build a system that behaves like a serious CTF operator?

That means it should:

  • ingest messy challenge inputs without manual cleanup
  • understand what kind of problem it is looking at
  • decide which workflow fits the problem
  • use specialized tools instead of pretending pure language is enough
  • conserve cost on easy wins
  • escalate intelligently on hard ones
  • recover when tools crash or models stall
  • remember what worked before
  • benchmark itself honestly
  • operate on local artifacts and on live platforms
  • produce logs, reports, replays, and artifacts that let you understand what happened afterward

Leaf Litter is my attempt to build exactly that.

I do not think of it as "a script that solves CTFs." I think of it more like a challenge operating system for CTF automation.

Why Squid Agent Mattered So Much

Squid Agent was the inflection point.

What made it compelling was not only the benchmark result. It was the design philosophy behind it:

  • category specialization instead of uniform prompting
  • tool discipline instead of tool sprawl
  • orchestration as a real system concern
  • benchmarking as the standard instead of anecdotes
  • the recognition that good CTF automation should look more like a specialized team than a monolithic chatbot

That framing immediately felt correct.

If strong human teams specialize by category, why would an automated system deliberately avoid specialization?

If hard challenges are often won by workflow quality, why would a solver rely only on one agent with a giant prompt?

If one of the biggest failure modes in CTF automation is burning time on the wrong plan, why would you not invest in routing, replay, recovery, and strategy selection?

Leaf Litter follows Squid Agent very directly on those principles. The overlap is intentional. Squid Agent showed that this design direction is real.

But I also wanted to push hard in places that many research-style writeups do not emphasize enough:

  • operator experience
  • environment readiness
  • fallback behavior
  • live platform automation
  • persistence and memory
  • replayability
  • runtime safety and recovery

That is where Leaf Litter became its own thing.

What Leaf Litter Actually Is

At a high level, Leaf Litter has five big layers:

  1. Challenge intake and normalization
  2. Fast-path and category-aware solving
  3. Self-healing runtime control
  4. Operator-facing control surfaces
  5. Benchmarking, platform mode, memory, and replay

The important part is that these layers are tightly connected.

The normalizer feeds the generator. The generator feeds the solver. The solver feeds memory, RAG, snapshots, and reports. The control center and platform mode sit on top of the same underlying workflow.

That means the system is not a loose collection of scripts. It is a coordinated stack.

The Design Philosophy

Before getting into implementation details, it helps to state the philosophy clearly.

1. Specialization beats fake universality

Leaf Litter does not pretend that rev, pwn, crypto, web, and forensics are the same problem with different labels.

Each category gets its own role graph, tool surfaces, model assignments, and pre-analysis behavior.

2. Tool curation matters more than raw tool count

One of the easiest ways to ruin an agent is to give it access to everything all the time.

Leaf Litter goes the other way. Tools are assigned by role and category. A pwn script-development role gets a different surface than a crypto critic role or a web exploit-development role. The system generator builds those lists automatically from the challenge category and inferred techniques.

3. Easy challenges should stay cheap

Not every challenge deserves the full solver stack. Some challenges are one obvious script run, one obvious decode, or one obvious file inspection away from a flag.

That is why Leaf Litter has a Codex-first fast path.

4. Hard challenges need recovery, not just persistence

Agents stall. Tools crash. Token windows blow up. APIs timeout. Responses get repetitive. Exploit attempts fail remotely while working locally. Good systems do not just keep going. Good systems adapt.

Leaf Litter bakes that adaptation into the runtime.

5. A solver should leave evidence behind

If the system solved something, it should produce logs, snapshots, reports, artifacts, and memory updates. If it failed, it should still teach the next run something useful.

Architecture Overview

At a system level, the pipeline looks like this:

Challenge Input
  -> Normalizer
  -> Normalized Descriptor + Files Workspace
  -> Capability Detection + Preflight + Solve-Time Estimation
  -> Codex-First Fast Path
  -> Category-Specific Preanalysis
  -> Generated Multi-Agent Runtime
  -> Strategy Ensemble / Replay / Escalation / Fallback
  -> Flag Resolution + Validation + Optional Platform Submission
  -> Memory / RAG / Snapshots / Reports / Logs / Artifacts

That flow is important because Leaf Litter is not "just agents."

It is:

  • an ingestion system
  • a planner
  • a runtime
  • a control plane
  • a reporting stack

Challenge Intake: Normalization as a First-Class System

One of the first places Leaf Litter becomes more serious than a typical toy solver is challenge ingestion.

CTF inputs are messy in practice:

  • a single binary
  • a folder with random names
  • a zip archive
  • a CTFd challenge payload
  • a JSON descriptor
  • some files plus a copied challenge description
  • maybe a service host and port
  • maybe not

Leaf Litter does not assume the inputs are already clean.

The normalizer:

  • accepts raw descriptors and local files
  • copies or extracts artifacts into a consistent ./files workspace
  • infers category when it is not explicitly given
  • infers likely entry points
  • infers initial techniques from keywords and files
  • writes standardized challenge metadata into ./normalized/challenge_env.json
  • writes an agent-facing descriptor into ./normalized/challenge_descriptor.json

That sounds mundane, but it matters tremendously.

This is the difference between an automation stack that works in ideal demos and one that can actually start from random challenge material without asking the operator to manually reshape it every time.

Leaf Litter even supports a mode where normalization can stay strictly local and fast, or optionally use Codex assistance when you want smarter interpretation.

That means the system is designed for real-world variance, not just curated benchmark inputs.

Generated Solvers, Not Static Solvers

One of my favorite parts of Leaf Litter is that it does not just load a fixed runtime and hope for the best.

It has a code generator that builds solver artifacts tailored to the normalized challenge.

That generator creates:

  • system prompts
  • curated tool lists
  • agent setup code
  • the single-challenge solver
  • the parallel solver

The generator is category-aware and role-aware.

For example:

  • rev gets a manager, binary-analysis role, debugger, and script-development role
  • pwn gets a manager, binary-analysis role, script developer, debugger, and code reviewer
  • crypto gets a manager, vulnerability analyst, critic, guessy fast-path agent, and script developer
  • web gets a manager, CWE analyst, vulnerability researcher, triage role, exploit developer, and script developer
  • forensics and misc get flatter role graphs, but still keep specialized structure

That alone is a major philosophical link to Squid Agent: the workflow graph depends on the challenge class.

But Leaf Litter adds another layer by generating the support wiring around those graphs:

  • model assignments per role
  • maximum steps per role
  • managed-agent hierarchies
  • final answer validation hooks
  • watchdog registration
  • token budget wiring
  • cost circuit breakers
  • model routing and escalation
  • even cross-system bridge agents for misc mode

So the generator is not just a convenience. It is the factory that converts a normalized challenge into an execution architecture.

Tool Discipline and Role-Specific Access

Leaf Litter has a lot of integrated tooling, but that is not the most important part.

The important part is how the tools are exposed.

The code generator curates tools per role and per category. That means:

  • a pwn manager sees a different tool universe than a pwn debugger
  • a crypto critic sees a different tool universe than a crypto guessy agent
  • a web CWE analyst sees a different tool universe than a web exploit developer

This makes the system much more coherent.

The roles are not just prompt flavors. They have different powers, different expectations, and different solution responsibilities.

That reflects a broader principle I took from Squid Agent and then pushed harder: strong tool discipline is part of agent design, not an implementation detail.

The Codex-First Fast Path

Leaf Litter does something that I think is extremely practical and honestly underrated: it gives the challenge a cheap direct shot first.

Before spinning up the full multi-agent system, Leaf Litter can build a tightly scoped Codex prompt from:

  • the challenge ID
  • category
  • description
  • inferred entry point
  • relevant files in the workspace

Then it tells Codex to do the obvious thing:

  • inspect the obvious files
  • run the obvious local script or binary when appropriate
  • stay within the workspace
  • print a flag only if it is actually confident
  • otherwise explicitly return unsolved and hand off

This is exactly the kind of feature that matters in practice:

  • cheap wins stay cheap
  • easy challenges do not burn the full stack
  • the solver feels responsive instead of always heavy

It also makes the whole system more honest. Instead of pretending every challenge is a deep orchestration problem, Leaf Litter acknowledges that some are trivial if you just take the direct path immediately.

The Small Features That Make It Feel Like a Real System

One thing I do not want to undersell is how many "small" systems in Leaf Litter end up mattering a lot in practice.

Individually, none of these features define the project. Collectively, they make it feel much more complete:

  • image entry points can go through a vision-analysis pass before the main solver starts
  • local artifact trees are pre-scanned for obvious flag candidates before expensive reasoning kicks in
  • Z3-style prechecks can opportunistically fire on constraint-heavy rev, crypto, or misc challenges
  • writeup hints can be scraped later in a run as untrusted suggestions instead of immediate crutches
  • platform submissions have confirmation, rejection, and rate-limit-aware backoff logic
  • flags can be validated before acceptance instead of being trusted just because an agent said so

This class of feature is easy to miss when you describe the project only in terms of "agents" and "categories."

But in real usage, this is exactly the kind of effort that separates a system that demos well from a system that feels carefully built.

Capability Detection, Preflight, and Solve-Time Estimation

Another part that makes Leaf Litter feel more like a full system than an experiment is what happens before the main solving loop begins.

It does not simply say "category = pwn" and start prompting.

It runs several planning layers first:

Capability detection

Leaf Litter scans the descriptor and file layout to infer reusable capability patterns such as:

  • binary exploit chain
  • crypto attack cycle
  • symbolic constraint solving
  • web chain building
  • forensics carving
  • jailbreak or sandbox escape
  • container surface auditing

Each capability has:

  • signals
  • recommended tools
  • a small playbook
  • a confidence score

That gives the solver a more structured sense of what kind of attack posture to adopt.

Preflight analysis

Leaf Litter builds a file inventory and extracts signals like:

  • presence of archives
  • image artifacts
  • network captures
  • binaries
  • web source
  • crypto materials
  • high-entropy files

From that it produces a suggested focus list before the agents even start.

That is a huge quality-of-life improvement because it biases the system toward the right early moves instead of making every run rediscover the same obvious facts.

Solve-time estimation

This is another very strong system feature: Leaf Litter tries to estimate how long a challenge may take using:

  • category priors
  • descriptor complexity
  • retrieved RAG similarity from prior solves

This is not just cosmetic. It feeds operator expectations and creates a more mature sense of solve planning.

A serious system should have opinions about likely difficulty and cost. Leaf Litter does.

Category-Specific Preanalysis

Before the multi-agent solve loop even begins, Leaf Litter can build category-specific context objects. This is one of the biggest reasons the system feels deep.

Reverse engineering preanalysis

For reversing challenges, Leaf Litter can:

  • detect anti-analysis behavior
  • patch anti-analysis obstacles
  • detect packers
  • unpack packed binaries
  • identify WASM workflows
  • build symbolic-analysis hints

That means the rev stack can start from an already-structured understanding of the binary rather than beginning every challenge from zero.

Pwn preanalysis

The pwn path is especially deep. It can:

  • run rev-style preanalysis first
  • detect mitigations
  • inspect container context and escape vectors
  • detect kernel challenge modes
  • analyze kernel modules
  • infer exploit primitives
  • propose exploit chains
  • generate pwntools chain skeletons
  • analyze heap layouts and generate heap-shaping snippets
  • build ROP chains
  • fall back to advanced techniques like ret2vsdo or ret2dlresolve
  • generate restricted-open-shellcode style plans under seccomp

This is not a shallow "maybe try buffer overflow" pipeline. It is a full exploit-planning surface.

Crypto preanalysis

For crypto, Leaf Litter tries to reason from the actual source and challenge text. It can probe for:

  • lattice-style attacks
  • multi-key RSA scenarios
  • PRNG weaknesses
  • ECB behavior
  • padding-oracle conditions
  • timing-oracle conditions

It also supports a strict four-stage cycle:

  • write
  • run
  • interpret
  • review

That is exactly the kind of discipline crypto challenges need.

Web preanalysis

The web stack is maybe the clearest example of why category specialization matters.

Leaf Litter performs a phased pipeline that can include:

  • error-trigger fingerprinting
  • source-map extraction
  • JS bundle analysis
  • websocket discovery
  • OAuth flow mapping
  • injection-site discovery
  • path traversal testing
  • request smuggling detection
  • SSTI detection
  • SQLi/NOSQLi/LDAP/XXE/deserialization checks
  • IDOR and mass-assignment checks
  • second-order bug scanning
  • session analysis and forged-session probes
  • CSRF evaluation
  • race-condition and TOCTOU testing
  • blind-oracle probing
  • DOM XSS and client-side traversal analysis

And when a multi-step exploit is needed, the stack can use a stateful exploit sequencer that carries session state and extracted values forward across requests.

That is a serious web exploitation pipeline, not just "send some requests and hope."

Forensics preanalysis

The forensics stack is also much deeper than a typical misc bucket.

Leaf Litter has a phased orchestrator that classifies artifacts and routes them through appropriate tool paths for:

  • images
  • audio
  • video
  • pcaps
  • disk images
  • memory dumps
  • archives
  • text artifacts
  • unknown blobs

It supports visual, barcode, LSB, palette, DCT, audio, QR, pcap, memory, disk, archive, and encoding-chain analysis. The orchestrator deepens only when necessary and can stop early if a high-confidence flag emerges.

Misc and cross-system bridging

The misc category is often where automation breaks down, because the challenge is either broad, weird, or badly labeled.

Leaf Litter handles that by allowing misc to bridge into other specialist systems. That is a strong idea because many misc problems are really hidden rev, pwn, crypto, or web challenges in disguise.

The Runtime Is Self-Healing

This is one of the biggest reasons I think Leaf Litter is more than a normal project.

A lot of agent systems are impressive until the first rough edge. Then they either crash, loop, or silently waste money.

Leaf Litter has multiple runtime systems specifically built to prevent that.

Agent watchdog recovery

The watchdog supervises agent execution and classifies failures such as:

  • API timeout
  • rate limit
  • context overflow
  • JSON parse error
  • subprocess crash
  • tool call failure

It can recover by:

  • backing off
  • waiting for retry windows
  • chunking context
  • forcing JSON-only responses
  • skipping broken tools
  • restarting subprocesses

That alone is a huge practical upgrade over naive orchestration.

Token budget management

Leaf Litter tracks token usage per agent and can intervene when context is getting too large.

It supports:

  • budget checks by model tier
  • warnings as context grows
  • chunking when needed
  • emergency summarization when nearing hard limits
  • session-wide token reporting

In other words, it does not simply discover context overflow by crashing into it.

Model routing and escalation

The model router watches for no-progress behavior by comparing successive agent outputs and looking for repetitive states without meaningful activity.

If an agent is stuck, it can escalate from cheaper tiers to stronger ones.

That gives Leaf Litter an answer to a common problem:

"This agent is still talking, but is it actually moving?"

Cost circuit breaker

Leaf Litter tracks:

  • per-challenge cost
  • global cost
  • warning thresholds
  • pause or abort conditions

That is important because serious multi-agent systems can burn budget extremely fast if they are not governed carefully.

Partial flag assembly

This is one of my favorite weirdly clever features.

Leaf Litter does not only look for complete flags. It also watches for fragments across tool outputs and agent responses, stores them, and attempts to assemble them later.

That matters a lot in categories like forensics, web, and crypto where the signal may leak out in pieces before the system sees the final string.

Human-in-the-loop stuck alerts

Leaf Litter can send stuck notifications through Discord, ask whether it should continue, and even accept injected hints from a human operator.

This is an important philosophical point:

Leaf Litter is trying to automate aggressively, but it is not built around the fantasy that humans should disappear entirely.

Instead, it lets a human step in at the right moments without manually taking over the entire solve.

Strategy Systems: Ensemble, Rotation, Replay

This is another area where Leaf Litter is much more advanced than the first draft made it sound.

It does not pick one plan and blindly grind.

It has:

  • category-specific strategy plans
  • profile selection like balanced, aggressive, and conservative
  • optional strategy ensembles
  • per-strategy budget allocation
  • historical category learning from prior logs
  • replay and strategy rotation when approaches fail
  • explicit exclusion of already-attempted strategies

That means the solver has a real concept of plan diversity.

It can move through strategy families like:

  • mitigation-first
  • dynamic trace
  • implementation-first
  • math-core
  • whitebox-priority
  • chain-hardening
  • deep carve
  • decode pipeline

And if those fail, it can rotate into replay stacks like:

  • angr_symbolic
  • heap_exploit
  • timing_oracle
  • graphql_attack
  • vision_analyze

This is very important. Many CTF failures are not due to lack of intelligence. They come from staying on the wrong strategy too long.

Leaf Litter is built to fight that failure mode directly.

Memory, RAG, and Cross-Challenge Learning

A strong CTF system should not reset its brain between runs.

Leaf Litter includes several persistence layers:

Session memory

It stores:

  • prior solves
  • failures
  • platform sessions
  • RAG updates

That lets the system look up similar prior solves and bias itself with evidence from actual history.

FAISS-backed RAG stores

Leaf Litter uses vector stores for:

  • rev
  • pwn
  • crypto
  • web
  • forensics
  • shared solve templates

Successful solves can update those stores, and retrieved entries can be scored and pruned over time.

This is crucial because CTF work is highly pattern-based. Many "new" challenges are really slight mutations of old ideas.

Artifact store

Leaf Litter also keeps a session-global artifact store that can register outputs and query them later by keywords or artifact type.

That enables cross-challenge context reuse rather than total isolation.

RAG-informed estimation and execution

The RAG layer is not just a passive note dump. It feeds:

  • solve-time estimation
  • guessy escalation
  • post-solve updates
  • retrieval-guided strategies

That makes the memory loop much tighter than a simple "save logs somewhere" approach.

Web Depth: A Full Exploit Research Stack

I want to call out the web subsystem specifically because it is one of the clearest examples of Leaf Litter being more ambitious than a normal agent project.

It has support for:

  • SQL injection
  • blind oracles
  • NOSQL injection
  • LDAP injection
  • SSTI
  • request smuggling
  • path traversal and LFI
  • XXE
  • deserialization
  • OAuth attacks
  • session analysis and forging
  • JWT attack suites
  • CSRF analysis and bypassing
  • SSRF probing
  • IDOR and mass assignment
  • race conditions and TOCTOU
  • websocket fuzzing and replay
  • source-map extraction
  • JS bundle analysis
  • DOM-side issue hunting
  • multi-step exploit sequencing

That is not "an agent that can try web things."

That is a category-specific offensive workflow framework exposed to an agent system.

Pwn Depth: From Mitigations to Container Escape

The pwn system is similarly deep and deserves special attention.

Leaf Litter is not limited to "run pwndbg and ask the model to write pwntools."

It explicitly reasons about:

  • mitigation profiles
  • exploit primitives
  • heap layout shaping
  • ROP construction
  • restricted-open-shellcode style fallbacks
  • libc inference and auto-patching
  • advanced ROP techniques
  • kernel challenge detection
  • kernel module analysis
  • containerization and breakout opportunities
  • fuzz-based fallback for crash-driven exploit discovery

That means the pwn path spans from classic userland exploitation into container escape and kernel-oriented workflows.

Again, that is the kind of breadth that makes this feel genuinely ambitious.

Forensics Depth: Phased Orchestration Instead of Misc Guessing

Another place where Leaf Litter gets a lot stronger than typical systems is forensics.

Many solvers treat forensics as an unstructured pile of scripts. Leaf Litter instead uses a phased orchestrator with file classification, ranked tool order, and early-stop logic.

It can walk through:

  • metadata extraction
  • hidden/appended data
  • archive cracking
  • text stego
  • visual stego
  • palette and DCT stego
  • audio spectrogram work
  • video analysis
  • pcap and wifi analysis
  • disk and memory forensics
  • carving and recursive extraction
  • generalized encoding-chain decoding

That matters because forensics is often won not by one magic technique, but by strong triage and the discipline to branch in the right order.

The Control Plane: Terminal UI and Control Center

This is one of the biggest ways Leaf Litter diverges from a lot of similar systems.

It is not just run from a raw command line with a long wall of flags.

It has:

A terminal UI

The terminal UI is lightweight and dependency-light on purpose. It is meant to work even when the environment is only partially healthy.

It gives quick access to:

  • health checks
  • feature audit
  • common-fix actions
  • dashboard launch
  • guided command building

A full browser control center

The control center is much more than a cosmetic frontend. It is a real operations console.

It lets you:

  • load local files or folders from Downloads
  • drag and drop challenge material
  • fill challenge metadata in plain English
  • run health checks and audits
  • launch single solves and platform solves
  • preview commands before launch
  • monitor background processes
  • inspect logs
  • view active challenge state
  • keep the workflow unified in one place

This matters because serious automation work is not only about the solver. It is about how quickly and confidently you can operate the solver.

That is one of the areas where I think Leaf Litter makes a very strong contribution.

Benchmark Mode and Parallel Solving

Leaf Litter is not only built for single local challenges.

It has a parallel benchmark path that:

  • spins isolated workspaces per challenge
  • normalizes each challenge
  • generates per-challenge solvers
  • runs them under time, cost, and round limits
  • aggregates category stats and termination reasons
  • produces benchmark reports in JSON and Markdown

That means the system is designed to be measured properly.

This is critical. Without benchmarking, agentic CTF projects become story generators. With benchmarking, they become engineering projects.

Platform Mode: Solving Live CTFd Queues

Leaf Litter also has a serious live-platform path.

It can:

  • log into a CTFd-style platform
  • list challenges
  • identify unsolved rows
  • download challenge attachments
  • normalize them automatically
  • generate per-challenge solvers
  • run challenges in parallel
  • optionally auto-submit flags
  • handle submission backoff
  • track costs and acceptance

And it does not stop there.

The platform mode also includes:

  • leaderboard tracking
  • challenge solve distribution awareness
  • dependency-aware scheduling
  • first-blood or low-solve prioritization
  • scoreboard tracking
  • session post-mortem reports

That is a big deal.

Most systems are either local solvers or live bots. Leaf Litter is trying to be both.

EMC Mode and Docker Networking

Leaf Litter also has a structured Docker networking layer.

For local and parallel runs, it can create challenge-specific Docker networks, and in EMC mode it creates per-challenge custom networks for cleaner separation.

This matters because realistic challenge environments often require:

  • controlled network assumptions
  • service reachability
  • isolated challenge contexts

In other words, Leaf Litter is aware that exploit automation often depends on environment shape, not only on reasoning.

Validation, Submission, and Trust

One of the easiest ways for automated solvers to embarrass themselves is by hallucinating or over-trusting weak flag candidates.

Leaf Litter addresses that with a fairly serious validation layer:

  • flag format validation
  • printable-character checks
  • entropy checks
  • optional local oracle validation
  • optional platform-side dry-run or submission logic
  • strict flag consensus mode
  • ranked candidate tracking

This is another subtle but important point:

Leaf Litter is trying to act like a trustworthy solver, not just a generative one.

Reports, Snapshots, and Reproducibility

This is one of the strongest system-design decisions in the whole project.

When Leaf Litter solves a challenge, it does not just say "flag found" and vanish.

It can also produce:

  • JSON run logs
  • Markdown run summaries
  • session post-mortem reports
  • challenge snapshots containing generated code, normalized state, files, logs, and command traces
  • replayable solve artifacts
  • token and watchdog reports
  • partial-flag assembly reports

That means Leaf Litter is useful even after the solve.

You can inspect what happened, replay pieces, compare strategies, and actually learn from the run.

That is a very different standard from most prototype agent systems.

How Similar Leaf Litter Is to Squid Agent

At the architectural level, Leaf Litter is absolutely in the same lineage as Squid Agent.

The strongest shared ideas are:

  • category-specific agent systems
  • multi-agent orchestration
  • curated tool usage
  • benchmarking as the standard for progress
  • the belief that specialization is the correct abstraction for CTF automation

If someone reads both projects side by side, the inspiration is obvious, and I do not think that is a bad thing. Squid Agent helped establish a direction that deserved to be built on.

Where Leaf Litter Tries to Go Further

Where Leaf Litter pushes beyond that inspiration is in the surrounding system design.

I wanted more than a category-aware solver. I wanted a solver with:

  • a fast-shot path for cheap wins
  • a real control plane
  • challenge normalization as a first-class concern
  • capability and preflight intelligence
  • memory and cross-run retrieval
  • self-healing runtime behavior
  • strategy rotation and replay
  • live platform solving
  • notifications and stuck handling
  • snapshots and post-mortems

So if I had to summarize the relationship in one sentence:

Squid Agent inspired the architecture; Leaf Litter turns that architecture into a broader operational system.

Why I Think Leaf Litter Is Actually Important

I do not think Leaf Litter is interesting merely because it has a lot of features.

Plenty of projects have a lot of features.

I think it is important because it pushes on a more meaningful question:

What does it look like when CTF automation stops being a novelty and starts becoming infrastructure?

Leaf Litter feels closer to that answer than most systems I have seen because it is not only about solving logic.

It is about:

  • environment readiness
  • system resilience
  • solve economics
  • operator control
  • memory
  • replay
  • live competition support

That is why I think it has the potential to be genuinely consequential.

Not because it is flashy. Because it is starting to behave like a real system.

What I Plan to Add Next

Leaf Litter is already much larger than I first intended, but it still feels like the beginning rather than the endpoint.

There are several major directions I want to push further.

1. Deeper per-category stacks

The category split is strong already, but I want each specialist track to become even more mature.

That means:

  • better rev workflows for larger and uglier binaries
  • stronger kernel and heap pwn automation
  • better crypto attack reformulation when classical assumptions fail
  • stronger web exploit confirmation and exploit-chain minimization
  • more robust misc triage into other subsystems

2. Stronger learning loops

The RAG and memory layers are a solid base, but I want:

  • better entry scoring
  • smarter promotion and pruning
  • stronger retrieval conditioning
  • more deliberate solve-template libraries
  • cross-challenge artifact reasoning that is less keyword-based and more semantic

3. Better large-challenge handling

One of the hardest problems in agentic CTF work is scale. Large binaries, large codebases, and large artifact sets can still overwhelm the system.

I want better:

  • context partitioning
  • long-horizon state retention
  • hierarchical tasking for large targets
  • guided subproblem decomposition

4. Better live competition behavior

Platform mode is already useful, but I want it to get much sharper in live settings:

  • better challenge triage
  • smarter dependency-aware queues
  • cleaner first-blood prioritization
  • stronger operator dashboards
  • improved rate-limit handling
  • better submission consensus logic

5. Better exploit and decompiler infrastructure

Tooling is still one of the main ceilings for hard challenges. I want stronger integration with:

  • reversing infrastructure
  • symbolic tooling
  • fuzzing and crash triage
  • kernel debugging workflows
  • exploit verification harnesses

6. Richer reporting and evaluation

The benchmark runner and post-mortem systems are already valuable, but I want more:

  • regression tracking across versions
  • clearer solve-delta comparisons
  • deeper failure clustering
  • stronger dataset-level analytics

7. Better human/agent collaboration

The Discord stuck-alert path is a good first step, but I want Leaf Litter to become better at asking for the right kind of help at the right time:

  • when to ask for a hint
  • when to ask for approval to keep spending
  • when to request a category pivot
  • when to expose an internal report to a human instead of blindly continuing

Final Thoughts

Squid Agent made one thing obvious to me: category-specialized CTF automation is not a gimmick. It is the correct direction.

Leaf Litter is what happened when I took that lesson seriously and kept pushing.

It is:

  • a challenge normalizer
  • a generator for role-specific solver graphs
  • a fast-path plus full-stack hybrid solver
  • a capability and preflight engine
  • a memory and RAG system
  • a strategy and replay system
  • a watchdog-protected runtime
  • a control center
  • a benchmark harness
  • a live CTF platform operator
  • a snapshot and reporting stack

That is why I think the interesting story is not simply "I built something like Squid Agent."

The real story is:

I took inspiration from Squid Agent and started building the kind of CTF system I actually wish existed: one that does not just solve challenges, but knows how to load them, plan them, adapt during them, recover from failure, remember what happened, and scale from a single local file to a live competitive queue.

It is very similar to Squid Agent in the best way possible: it inherits the right big idea.

But it is also increasingly its own thing.

And honestly, that is what makes Leaf Litter exciting to me.

It does not feel like a finished project. It feels like the first version of a system that could seriously change how people think about CTF automation.
