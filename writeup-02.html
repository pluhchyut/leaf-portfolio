Case Study 02 • Binary Exploitation

Router: turning leaked output and bad indices into a heap exploit

Flag recovered • adapted from the five-case credibility essay
Binary Exploitation Heap No Source OOB Index Case Study

Challenge Summary

This challenge looked like a harmless route planner on the surface. The real work was learning how orders, maps, and rerouting logic were arranged on the heap when the only interface was a text menu.

Why It Was Hard

There was no source, the binary changed per instance, and the bug only surfaced through user-facing indices. The useful hints were behavioral, not structural: strange output and indices that pushed across object boundaries without immediately crashing the process.

Approach

  • Learn the normal program flow first. I used help, loaded maps, created orders, rerouted them normally, and dispatched them so I could see what healthy output looked like before touching the bug.
  • Use order creation as heap grooming. Multiple live orders forced the binary to allocate a repeatable set of objects, which made it easier to watch when route data started leaking neighboring state.
  • Probe the index boundary carefully. I moved from valid indices to edge cases and then to negative values. The goal was not a crash, but a leak that exposed internal state without killing the session.
  • Turn the leak into orientation. Once output started including bytes that did not belong to the selected order, I repeated the same allocation pattern until I could estimate which objects sat next to each other on the heap.
  • Promote the bug into control. After the layout stabilized, the out-of-bounds access stopped being “weird output” and became a constrained write into adjacent metadata, which was enough to redirect execution toward the hidden success path.

This mattered because the exploit path became repeatable. The binary taught me its own memory model through repeated interaction, and the final solve came from using that model instead of gambling on one-off corruption.

Terminal Snapshot

$ nc mysterious-sea.picoctf.net 51703
router> help
router> load city1.map
router> add_order A1 B7
router> add_order C3 D9
router> reroute -1
debug-ish output leaks internal route state
router> dispatch

Working theory: a negative or oversized index crosses from one heap object into the next
Exploit goal: use the leak to stabilize the corruption instead of brute forcing it

Outcome

Flag recovered in the original solve after stabilizing the out-of-bounds heap behavior with leaked state from the text interface. The important part of the writeup is that the exploit path became deliberate, not lucky.

When source is missing, the interface becomes the debugger. This case is in the archive because it shows how I build a memory model from behavior instead of guessing from vibes.
